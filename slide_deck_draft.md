## スライド 4: 課題の定義
**[Layout / Visual]**
- **背景**: [White #FFFFFF]
- **構成**: 対比図
  - 左：「消費財 (SNS)」薄いグレーで表現。アイコン：流れるタイムライン。
  - 右：「資産 (Web)」[Deep Navy]のボックスで強調。アイコン：積み上がるブロック。

**[On-Slide Text]**
> **3. 課題の定義**
>
> **作品は「消費財」ではなく「資産」である**
>
> *   **現状の課題 (SNS/Webサービス)**
>     *   タイムラインで流れて消える（フロー型）
>     *   サービス終了のリスク
>     *   データが手元に残らない
>
> *   **あるべき姿 (本研究の提案)**
>     *   **「ストック型」の管理**
>     *   自分の手元にデータを蓄積する
>     *   必要な時に、必要な形で取り出せる

**[Speaker Notes]**
「現状、多くのクリエイターはSNSで作品を発表していますが、それはタイムラインと共に流れて消えてしまう『消費財』のような状態です。
しかし、本来、作品とは積み上げていくべき『個人の資産』であるはずです。
既存のツールでは難しかった、この『作品の資産化』こそが、解決すべき最大の課題です。」

## スライド 5: 既存サービスとの比較
**[Layout / Visual]**
- **背景**: [White #FFFFFF]
- **構成**: 左右2カラム
  - 左：[Soft Red #FEF2F2] 背景の「課題」エリア
  - 右：[Soft Blue #EFF6FF] 背景の「解決策」エリア

**[On-Slide Text]**
> **4. アプローチ**
>
> **[課題] 導入のハードル**
> *   **WordPress (従来型)**
>     *   管理と表示がセットで、デザイン変更が重い
>     *   管理画面が使いにくい
> *   **SNS**
>     *   データが散逸する
>
> **[解決策] Headless CMSアプローチ**
> *   **管理 (WordPress)**
>     *   データの所有権
> *   **表示 (Next.js)**
>     *   **分離のメリット**
>     *   デザインだけ着せ替え可能
>     *   資産性（WP）と柔軟性（Next.js）の両立

**[Speaker Notes]**
「では、どうやってこれを実現したのか。
その解が『ヘッドレスCMS』というアプローチです。
従来、WordPressはデータ管理には最適ですが、デザイン変更が難しく、操作も複雑でした。
本研究では、『管理機能（WordPress）』と『表示機能（Next.js）』を完全に分離しました。
これにより、データの資産性を保ちながら、表現の自由度と最新のパフォーマンスを手に入れました。」

ここで単に画像を貼るだけでなく、制作プロセスやコンセプトといった『文脈（ナラティブ）』を記述できる構造こそが、評価者が本当に見たい情報だと定義しました。」

## スライド 7: デザインの方向性と立ち位置
**[Layout / Visual]**
- **背景**: [White #FFFFFF]
- **構成**: 4象限マトリクス図
  - 縦軸：情報量（Min <-> Max）
  - 横軸：体験（Static <-> Interactive）
  - **4つのタイプ**:
    1.  **Type A: 没入型** (Immersive) - 余白重視
    2.  **Type B: 文脈重視型** (Narrative) - 読み物
    3.  **Type C: カタログ型** (Archive) - 検索・量
    4.  **Type D: インタラクティブ型** - 演出重視
  - **本システムの領域**: Type BとCの重複領域を強調。
- **Assets**:
  ![Positioning Matrix](/Users/imac2015/.gemini/antigravity/brain/a1d7643e-aba8-4118-b036-40ba773d720d/diagram_positioning_matrix_1771177943367.png)

**[On-Slide Text]**
> **6. デザインの方向性 (Positioning)**
>
> **採用方針: 「Type B (Narrative)」+「Type C (Archive)」**
>
> *   **理由**:
>     *   学生の評価軸は一つではない。
>     *   **「思考の深さ（文脈）」** と **「制作の量と幅（アーカイブ）」**
>     *   この両方を同時に提示できるハイブリッド型を標準とする。

**[Speaker Notes]**
「次に、デザインの方向性です。
ポートフォリオサイトを分析すると、大きく4つのタイプに分類できました。
本システムでは、この中の『Type B：文脈重視型』と『Type C：カタログ型』のハイブリッドを採用しました。
なぜなら、美大生の就職活動や進学審査では、『どれだけ深く考えたか（思考の深さ）』と『どれだけ作ったか（制作量）』の両方が評価されるからです。
この両方を妥協せずに満たすことが、テンプレートとしての正解であると考えました。」

## スライド 9: 提案手法 (コンテキスト指向UI)
**[Layout / Visual]**
- **背景**: [Deep Navy #182238]
- **コンセプト**: "Prism (プリズム)"
- **構成**:
  - 左から入る「一本の光（WordPress）」が、中央のプリズム（API）を通り、右側に「美しいスペクトル（Mobile/Desktop/Print...）」として分光される。
  - グラスモーフィズム調の透明感あるビジュアル。

**[On-Slide Text]**
> **8. 提案手法：コンテキスト指向UI**
>
> **One Source, Multi Use**
> 「一つのデータから、相手や状況に合わせて最適な形へ」
>
> 1.  **Mobile**: 名刺代わり（自己紹介）
> 2.  **Desktop**: 就活・審査（高画質鑑賞）
> 3.  **Print**: ポートフォリオ提出（PDF）
> 4.  **Signage**: 学内展示モニター

**[Speaker Notes]**
「システムの核となる思想が、『コンテキスト指向UI』です。
この図のように、WordPressにある『一つのデータ』が、まるでプリズムを通すように、相手や状況に合わせて最適な形に変化して出力されます。
スマホなら名刺代わりに、PCならプレゼン用に、紙が必要なら印刷用に。
データさえあれば、出力先は無限に広がります。」

  ![Editor UI](/Users/imac2015/.gemini/antigravity/brain/a1d7643e-aba8-4118-b036-40ba773d720d/slide_asset_editor_ui_1771177726921.png)

**[On-Slide Text]**
> **9. システム構成とユーザー体験**
>
> **[Editor (React SPA)]** → **[API/WP]** → **[Next.js]** → **[User]**
>
> *   **Seamless Experience**
>     *   裏側の複雑さを意識させない専用エディタ
> *   **Performance (Next.js + CDN)**
>     *   世界中どこからでも一瞬で表示
> *   **Sustainable**
>     *   画像の自動最適化（高画質維持・通信量削減）

**[Speaker Notes]**
「システム構成も、ユーザー体験を最優先に設計しました。
ユーザーは、裏側の複雑な仕組みを一切気にする必要はありません。
専用に開発したエディタから作品を投稿するだけで、自動的に最適化され、世界中に高速配信されます。
『投稿』から『公開』までを、極めてシームレスにつなぎことで、継続的な運用を可能にしました。」
